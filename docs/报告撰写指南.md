# 报告撰写指南

本文用于把当前仓库事实整理成课程设计报告。功能、接口和测试结论应以 [`index.md`](index.md) 指向的权威文档为准。

## 1. 推荐题目与结构

```text
基于 RAG 的 Windows 本地课程资料库与 AI 学习助手系统设计与实现
```

推荐九章：封面、目的要求、可行性分析、需求分析、概要设计、详细设计、程序模块、测试与用户手册、个人体会。

## 2. 目的要求

应覆盖：

- 按本地一级文件夹自动识别课程。
- 浏览和预览 PDF、图片、TXT、Markdown，提取 DOCX 基础正文。
- 为每门课程建立独立 hybrid RAG 索引并返回可追溯引用。
- 在同一课程下管理多个独立对话和记忆。
- 提供答疑、启发提示、复习三种学习模式。
- 生成课程摘要、练习题并维护课程笔记。

不要把已移除的学习计划 UI/API 写成当前能力，也不要声称当前前端存在 Dashboard/掌握度操作区。

## 3. 可行性分析

- 技术：Python 标准库 HTTP 服务、Vue 3、TypeScript、Pinia、JSON 文件、pypdf 和可选 OpenAI-compatible/MinerU/MCP 接口。
- 经济：不需要独立数据库或向量数据库，可使用本地 fallback；外部模型可能产生费用。
- 操作：用户设置资料目录后即可按课程、对话和文件工作。
- 安全：默认仅监听本机；真实密钥不入库，外部服务数据边界需要明确说明。
- 环境：完整界面需要 Python、Node.js 和 npm，不能写成“无 Node.js 依赖”。

## 4. 需求分析

功能需求：资料目录、课程扫描、文件预览/搜索、知识库构建、多对话、三模式问答、ReAct 工具决策、流式 Markdown、引用、附件、摘要、练习题和笔记。

非功能需求：课程/对话隔离、路径安全、配置安全、流式可见性、外部服务失败诊断、跨平台安装和可回归测试。

需求图至少包括：

- 从资料目录到索引和引用回答的数据流图。
- 多对话与课程共享数据的边界。
- ReAct planner、工具 observation 和 responder 流程。

## 5. 概要设计

```text
Vue 工作台
  ├─ 课程 / 对话 / 文件树
  ├─ 三模式聊天与课程笔记
  └─ 资料预览
        │ JSON / upload / SSE / NDJSON
Python API
  ├─ api/chat + api/course + api/server
  ├─ retrieval + learning
  ├─ parser + llm + web
  └─ store + storage + ops
        │
本地课程文件 + data/course_memory + data/indexes
```

接口说明至少包含 `/api/config`、课程/文件/索引接口，以及 `/api/courses/{course_id}/conversations/...` 的对话级消息、memory、chat、summary、quiz 和 clear 路由。

## 6. 详细设计

E-R 图实体建议：Course、FileNode、DocumentChunk、VectorDocument、Conversation、ChatMessage、ConversationMemory、Citation、Note、MasteryState、StudyArtifact。

关键流程：

1. 课程扫描与路径过滤。
2. 文档解析、质量评估、切片和 lexical/vector 索引。
3. 当前对话历史和追问改写。
4. 三模式 ReAct planner、课程/联网工具、observation、responder。
5. 流式事件、引用检查、消息和 memory 持久化。
6. 旧单会话数据迁移。
7. 备份白名单和恢复路径校验。

## 7. 程序模块

代码摘录使用 [`程序模块与代码摘录.md`](程序模块与代码摘录.md)，优先展示 `CourseScanner.scan()`、`ChatFlow.run()` 的 ReAct 分支、conversation migration 和引用后处理。每段控制在半页以内，并注明真实源码路径。

## 8. 测试与用户手册

测试结论必须来自实际命令。建议表格至少覆盖：

| 类别 | 重点 |
| --- | --- |
| 后端单测 | 路由、存储、RAG、解析、摘要、备份、重试 |
| 前端单测 | 多对话、三模式、流事件、Markdown、停止、笔记、预览 |
| 静态检查 | TypeScript typecheck 和生产构建 |
| RAG 门禁 | demo baseline 的 retrieval、ChatFlow、summary |
| 浏览器验收 | 当前 UI、真实交互和 5 张截图 |

局部 `test_materials` 结果不能替代全系统验收。报告应写明日期、commit、命令、通过数量和外部服务未覆盖项。

用户手册以 [`使用说明.md`](使用说明.md) 和 `docs/screenshots/` 为准。

## 9. 个人体会

围绕真实工程取舍展开：RAG 证据约束、ReAct 工具决策、流式体验、多对话迁移、本地文件型存储、外部服务边界和自动化回归。限制应写复杂文档解析、外部 provider 质量、单机部署和当前未暴露的后端学习 API，而不是已经解决的旧问题。

- 图表编号、标题层级、页码和参考文献格式统一。
