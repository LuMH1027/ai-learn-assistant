# 个人体会

本项目最终形成了一个可在本机运行的课程资料工作台：按一级文件夹识别课程，预览 PDF、图片、TXT 和 Markdown，为每门课程建立独立 RAG 索引，并在同一课程下管理多个相互隔离的对话。问答不仅返回答案，还保留本地或网页引用、处理 trace、实时思考状态和课程级笔记，使学生能够回到原始资料复核。

实现中最重要的认识是，课程助手的可信度来自整条链路，而不是单个 Prompt。文档解析质量、标题感知切片、BM25/向量多路召回、RRF、rerank、MMR、引用后处理和会话上下文都会改变最终结果。ReAct planner 进一步把“是否查课程、是否联网、是否先澄清”变成显式决策；答疑、启发提示和复习三种模式同时约束工具选择与回答方式，比简单添加模式前缀更可靠。

本地部署也需要清楚区分“轻量”和“零依赖”。项目不需要 MySQL、Redis 或独立向量数据库，但完整界面必须使用 Node.js/npm 构建 Vue 前端，Python 服务负责 API 与静态文件。对话、记忆、笔记和掌握度采用 JSON/Markdown 文件持久化，便于查看和备份；多对话迁移又说明文件型存储同样需要版本兼容、路径安全和明确的数据契约。

当前仍有边界：DOCX 只支持基础正文，扫描文档依赖 OCR/MinerU；真实语义召回依赖外部 embedding，配置模型后的持续网络错误会直接反馈给用户；Dashboard 与 mastery 虽有后端 API，当前前端没有操作区。后续可以加强 OCR、multi-vector 检索、自动化端到端评测，以及把后端学习状态重新设计成更清晰的学生工作流。

这次课程设计让我更具体地理解了 RAG、流式交互和本地数据治理之间的关系。一个可用的学习助手既要回答得有依据，也要让用户看得见系统正在做什么、知道数据保存在哪里，并在外部服务失败时给出诚实而可诊断的状态。
