# 开发问题与解决记录

本文记录本项目从原型、UI 重设计、Vue 迁移、RAG 增强、联网检索到流式回答期间实际遇到的主要问题。每项均说明现象、根因、解决方式、验证方法和后续预防措施，可作为课程设计报告中的“问题分析与调试”章节，也可用于后续维护排障。

## 1. 需求从“资料管理”逐步收敛为“学生学习工作台”

**现象**

早期界面更像文件管理器或通用聊天页，课程、资料预览和学习对话之间缺少明确主次。继续堆按钮虽然能增加功能，但学生完成“选择课程、阅读资料、提问、核对引用”的路径仍然很绕。

**根因**

最初只明确了“把课程文件放进去后可以提问”，没有先固定首要用户、核心任务和高频操作。文件管理、课程 RAG、笔记、摘要、练习题都重要，但它们不应该在视觉上拥有相同优先级。

**解决**

- 将目标用户固定为日常学习的学生，而不是资料管理员。
- 将主流程固定为“课程选择 -> 资料阅读 -> 基于资料提问 -> 点击引用核对”。
- 左栏只承担课程和资料组织，中栏承担学习对话，右栏承担可开关的原文预览。
- 摘要、练习题和笔记保留为学习工具，不占据主对话的首要位置。

**验证与预防**

验收时不只检查按钮是否存在，而是完整走一次学生任务。后续新增功能前先判断它属于课程导航、对话、预览还是学习工具，避免重新形成无层次的功能堆积。

## 2. 原生前端难以继续扩展

**现象**

原生 HTML、CSS 和 JavaScript 可以快速做出原型，但随着课程切换、文件预览、附件、引用、笔记、三栏拖动和多种异步请求加入，状态散落在 DOM 和全局变量中。修改一处界面容易影响另一处，测试也只能依赖大量选择器字符串。

**根因**

原生实现缺少明确的组件边界和集中状态管理。课程、聊天、预览、布局分别有独立生命周期，却由同一批页面级函数共同维护。

**解决**

- 迁移到 Vue 3、TypeScript、Vite 和 Pinia。
- 将 UI 拆为 `CourseSidebar`、`ChatWorkspace`、`FilePreview`、`ResizableWorkspace` 和 `NotesDrawer`。
- 将课程、聊天、预览、布局分别放入独立 Store。
- 在 `services/api.ts` 统一处理 JSON、FormData、错误和流式响应。
- 保持 Python API 契约不变，先迁移前端边界，再替换页面，降低一次性重构风险。

**验证与预防**

使用组件测试、Store 测试、API 测试和 TypeScript 类型检查共同约束。生产环境只服务 Vite 构建产物，避免旧原生页面与 Vue 页面长期并存、行为漂移。

## 3. UI 视觉风格与学习场景不匹配

**现象**

早期 UI 使用大面积黑色侧栏、较重的容器和不统一的颜色，视觉压迫感强。用户希望参考 Codex 的安静工作台风格，同时需要比较深浅不同的侧栏配色。

**根因**

界面缺少统一的设计令牌，颜色、间距、边框和状态样式直接写在具体组件中。只替换某个背景色无法建立完整层级。

**解决**

- 建立背景、表面、边框、正文、弱化文字、强调色和错误色等 CSS 变量。
- 采用中性工作台配色，减少高饱和大面积颜色。
- 用边框、留白和文字层级组织信息，避免卡片套卡片。
- 保留清晰的焦点态、选中态、悬停态和错误文字，不能只依赖颜色表达状态。

**验证与预防**

通过样式契约测试检查关键设计变量和布局约束；在桌面与窄窗口检查文本溢出、按钮可点击范围和三栏信息密度。

## 4. 三栏拖动错误地影响了不相邻区域

**现象**

拖动左侧分隔线时，右侧预览区也会改变；拖动右侧时，左栏也会移动。用户要求总体宽度不变，并且每次拖动只交换分隔线两边的空间。

**根因**

最初按三个独立宽度重新归一化，任何一栏变化都会触发全局比例重算，没有把分隔线视为两个相邻区域之间的局部约束。

**解决**

- 使用百分比保存栏宽，使布局随窗口变化。
- 左分隔线只保持“左栏 + 中栏”的和不变；右分隔线只保持“中栏 + 预览栏”的和不变。
- 为每栏定义最小比例，并由布局 Store 集中裁剪。
- 双击分隔线恢复默认比例，键盘方向键也可以微调。

**验证与预防**

Store 测试分别记录拖动前后的三个比例，断言非相邻栏完全不变、三栏总和不变。布局算法不得直接依赖当前 DOM 像素值作为持久状态。

## 5. 紧凑侧栏的下限不能使用固定像素

**现象**

固定 `px` 下限在大屏上显得浪费，在小屏上又可能挡住中栏。用户要求以百分比调整，并在紧凑模式下尽量收缩到略大于栏内最大控件。

**根因**

固定像素没有表达“相对于工作区”的约束，也没有区分普通模式和紧凑模式。

**解决**

- 持久宽度统一使用百分比。
- 普通模式和紧凑模式采用不同最小比例。
- 进入紧凑模式时隐藏次要文字，只保留仍可识别的图标和关键控件。
- 使用 `minmax`、`min-width: 0` 和溢出处理，防止长课程名撑破布局。

**验证与预防**

测试窄、中、宽三种窗口，并验证本地存储中的旧布局值经过裁剪后仍可使用。

## 6. 关闭资料预览后中栏没有填满空间

**现象**

右侧预览关闭后，页面留下空白，聊天区没有扩大；重新打开时比例也可能丢失。

**根因**

预览“不可见”和预览“宽度为零”被当成两套互不关联的状态，网格列仍为关闭前的三栏定义。

**解决**

- 将预览开关作为布局状态的一部分。
- 关闭时移除预览列与对应分隔线，中栏自动占用剩余宽度。
- 保存关闭前的预览比例，重新打开时恢复，而不是使用硬编码宽度。

**验证与预防**

组件测试断言关闭预览后只存在两栏布局，中栏扩展；再次打开后仍回到可拖动的三栏。

## 7. 窗口变化会破坏已经保存的布局

**现象**

窗口缩窄后，原来合法的栏宽可能低于新窗口的可用下限；恢复窗口时布局还会跳动。浏览器禁用或损坏 `localStorage` 时，页面可能初始化失败。

**根因**

保存的是用户偏好，但渲染时没有重新应用当前窗口约束；同时把本地存储误当作永远可用的可靠数据库。

**解决**

- 持久化比例而非像素。
- 每次读取和窗口变化时重新裁剪到合法范围。
- 对 `localStorage` 读取、JSON 解析和写入全部容错，失败时使用内置默认值。
- 布局重置只恢复相关区域，不错误改变中栏总宽度。

**验证与预防**

覆盖无存储、非法 JSON、过期值和窗口缩放测试。任何持久化 UI 偏好都必须有默认值和版本兼容策略。

## 8. 快速切换课程时旧请求覆盖新页面

**现象**

用户点击课程 A 后立即切换到课程 B，如果 A 的网络请求较晚完成，页面可能重新显示 A 的文件树、消息或笔记。类似问题也发生在快速点击两个引用时，较慢的旧预览会覆盖新预览。

**根因**

异步响应只在发出时知道所属课程，完成时没有再次确认当前上下文。HTTP 返回顺序不能保证与用户点击顺序一致。

**解决**

- 为课程列表、课程树、消息、预览分别维护递增请求令牌。
- 发出请求时记录课程 ID、上下文版本和令牌。
- 响应落地前同时检查当前课程与令牌，过期响应直接丢弃。
- 根目录刷新与当前课程刷新使用独立令牌，避免互相取消。

**验证与预防**

Store 测试用可控 Promise 反转响应顺序，断言旧响应无法覆盖新状态。所有“选择变化后请求详情”的功能都应采用相同模式。

## 9. 配置和笔记的并发写入会丢数据

**现象**

多个操作几乎同时保存时，后完成的旧快照可能覆盖先完成的新值。界面重新加载后会发现部分设置或笔记消失。

**根因**

前端乐观状态与后端文件型权威状态混用，并发写操作没有串行化。读取旧对象、修改后整文件覆盖会造成 lost update。

**解决**

- 将配置和笔记写入视为权威操作，按 Store 串行提交。
- 保存成功后采用服务端返回值，不用旧前端快照覆盖。
- 不把配置刷新与课程数据刷新绑定在同一个请求生命周期中。

**验证与预防**

测试连续触发多次保存并反转响应时序，确认最终状态等于最后一次用户操作。新增文件型写操作时必须明确串行策略或冲突合并策略。

## 10. Vue 构建产物与 Python 静态服务不一致

**现象**

直接运行 Python 时可能看到旧页面，或因为 `web/dist` 不存在而得到空白页。开发服务器正常并不代表生产静态服务正常。

**根因**

Vite 开发模式从内存提供资源，而 Python 生产模式只能读取已构建文件。若同时保留旧静态入口，还会出现两个前端实现漂移。

**解决**

- Vite 固定输出到 `web/dist`。
- Python 只服务该目录，不回退到旧页面。
- `start.sh` 和 `start.bat` 在启动 Python 前强制执行类型检查和生产构建。
- 构建目录不提交 Git，避免平台相关产物污染仓库。

**验证与预防**

静态服务测试检查根路径、资源路径、目录穿越和缺失构建提示；启动契约测试确保构建命令一定早于 Python 服务。

## 11. Node 版本满足“20”但仍无法运行 Vite

**现象**

部分机器安装了 Node 20 的早期小版本，`npm ci` 可以执行，但 Vite 构建报运行时版本不兼容。只检查主版本会误判环境可用。

**根因**

当前 Vite 工具链要求 Node.js `20.19+` 或 `22.12+`，并非任意 Node 20/22。

**解决**

- 在 `frontend/package.json` 的 `engines` 声明精确范围。
- 所有安装和启动脚本执行相同的 Node 版本判断。
- 错误信息直接给出 `20.19+ or 22.12+`，避免用户只反复重装 npm 包。

**验证与预防**

启动契约测试检查两个版本边界同时出现在 shell、batch、README 和安装器中。升级 Vite 前先确认并同步运行时要求。

## 12. 只安装 pip 依赖无法启动完整项目

**现象**

早期文档重点说明 `pip install -r requirements.txt`，但迁移到 Vue 后还需要 npm 依赖。新用户只安装 Python 包后运行服务，会缺少 `web/dist` 或前端构建工具。

**根因**

项目从纯 Python 原型演变为 Python + Node 双工具链，依赖安装文档和脚本没有同步升级。

**解决**

- 新增 `install-deps.sh` 和 `install-deps.bat`。
- 自动创建 `.venv`，将 Python 包隔离在项目内。
- 使用 `requirements.txt` 安装 Python 包，使用 `package-lock.json` 驱动 `npm ci` 安装前端及测试依赖。
- 只有 pip 与 npm 全部成功后才写入 `.venv/.course-agent-deps-ready`；中途失败留下的 `.venv` 不会被启动脚本误判为完整环境。
- 启动脚本检测 `.venv` 或 `node_modules` 缺失时自动调用安装器。
- 不自动下载或替换系统 Python/Node，避免管理员权限、安全和平台包管理器差异；缺少运行时时给出可操作提示。

**验证与预防**

对 shell 脚本执行语法检查，测试安装器必须同时包含 pip、npm、锁文件、虚拟环境和版本检查。新增依赖后只修改对应清单并更新锁文件，不在脚本中散落包名。

### 补充：安装时出现 npm deprecated 警告

实际执行 `npm ci` 时，个别测试工具的传递依赖可能输出 `deprecated` 或安全维护提示。这类警告与命令非零退出不同：安装器只有在 `npm ci` 返回失败状态时才停止。处理时先用 `npm ls --prefix frontend --depth=0` 确认顶层依赖完整，再通过升级直接依赖和 lockfile 消化上游修复，不应绕过锁文件随意删除间接包。

## 13. 配置真实密钥容易被提交

**现象**

LLM、MinerU 和 Web Search 都可能需要 token。若直接编辑仓库中的示例配置，真实密钥可能进入 Git 历史，之后仅删除文件也无法彻底消除泄露。

**根因**

示例配置与运行配置没有明确分离，或者 `.gitignore` 规则过宽/过窄。

**解决**

- 只提交脱敏的 `data/config.example.json`。
- 真实 `data/config.json` 和生成的会话、索引、上传全部忽略。
- README 明确复制示例后再填写本地配置。
- 前端状态接口只返回“是否已配置”，不回传 API Key。

**验证与预防**

提交前检查 `git status` 和暂存区文件列表。若密钥曾进入提交，应立即吊销并更换，不能只依赖后续删除。

## 14. 配置了 LLM，但某些问题没有调用模型

**现象**

用户发现当前对话只是返回本地拼接结果，并未调用已配置的大模型。尤其在本地检索没有命中时，系统会直接给出固定降级文本。

**根因**

原路由把“存在课程引用”错误地当成“允许调用 LLM”的前置条件。本地没有证据时，模型调用也被一并跳过。

**解决**

- 只要模型已配置，就执行答案综合。
- Prompt 明确课程资料优先；资料未覆盖时允许标注后使用模型通用知识。
- 将 `used`、`fallback`、`disabled` 写入处理轨迹，便于确认是否真正调用。
- 模型失败时保留本地回答，不让整个问答接口不可用。

**验证与预防**

后端测试覆盖“无本地引用但模型已配置”场景，并断言模型仍被调用。路由条件必须围绕能力是否配置，而不是检索结果是否理想。

## 15. 中文课程检索命中率和排序不稳定

**现象**

中文没有天然空格，简单按词切分会漏掉“虚拟内存”“二叉搜索树”等术语。重复出现同一关键词的长段落还可能获得不合理高分，短数字问题会误匹配列表序号。

**根因**

早期轻量关键词评分没有文档长度归一化、中文子词、查询质量判断和噪声过滤。

**解决**

- 使用中文词项与字符 n-gram 共同表示文本。
- 使用 BM25 进行长度归一化和词频饱和，避免重复词无限加分。
- 过滤过短数字版本片段和通用问句词，减少列表编号误命中。
- 将检索质量区分为充分、较弱和无结果，供后续联网决策使用。

**验证与预防**

测试中文术语、重复词、数字查询、未知问题和课程隔离。每次调整分词或权重都应保留一组固定课程问答作为回归集。

## 16. 单一路径检索容易返回重复片段

**现象**

排名靠前的多个结果可能来自同一文件的相邻位置，虽然分数高，但答案缺少其他资料视角；截取单个片段还可能丢失定义前后的上下文。

**根因**

只按单一相关性分数排序，没有兼顾结果多样性，也没有将相邻块恢复为阅读上下文。

**解决**

- 多路检索结果通过 RRF 融合，降低某一路径偶然偏差。
- 使用 MMR 在相关性与多样性之间取舍，减少近重复结果。
- 对命中片段扩展相邻上下文，但保留原始来源和页码。
- 引用统一编号，回答可以追溯到课程文件。

**验证与预防**

测试结果必须来自多个来源且不过度重复，并断言相邻上下文扩展不会跨文件。

## 17. AI 生成摘要和练习题会污染课程检索

**现象**

生成的摘要、练习题被保存回课程目录后，下一次构建知识库又把这些 AI 内容作为课程依据。多轮之后会形成“模型引用模型”的反馈循环，原始课程资料权重下降。

**根因**

文件扫描只按扩展名判断是否可索引，没有区分教师/学生原始资料与系统派生产物。

**解决**

- 对 `AI生成/` 目录和历史生成文件建立明确排除规则。
- 生成产物仍保留在文件树中供学生阅读，但不进入课程证据索引。
- 兼容旧版本产生的文件名，避免升级后重新污染。

**验证与预防**

测试生成摘要和练习题后重新构建索引，断言生成文件可见但不成为引用。未来新增派生产物类型必须同步更新排除规则。

## 18. 联网搜索既不能每次调用，也不能完全禁用

**现象**

每个问题都联网会增加延迟、成本和隐私暴露，并可能让低质量网页盖过课程资料；完全不联网又无法回答最新信息或课程缺失内容。

**根因**

缺少“是否需要联网”的决策层，也没有区分课程来源与网页来源。

**解决**

- 将联网决策收敛到 ReAct planner；planner 可选择 `final`、澄清、课程检索、联网或组合工具，`final` 后仍由 responder 生成最终回答。
- 把课程/网页工具结果写成 observation，最多运行三轮；同类工具不重复调用。
- 只把学生问题发送给 MCP，不发送课程正文和附件内容。
- 本地来源使用 `[L1]`，网页来源使用 `[W1]`，界面中的网页引用可点击。
- 搜索失败、未配置或无结果都会写入处理轨迹，responder 再依据已有 observation 作答。

**验证与预防**

分别测试本地充分、本地不足、明确最新信息、MCP 无结果和网络失败。联网策略变化必须同时检查隐私边界。

## 19. MCP 搜索服务可能返回 JSON 或 SSE

**现象**

不同 Streamable HTTP MCP 服务的初始化、工具发现和调用返回格式不完全一致。有的直接返回 JSON-RPC JSON，有的通过 SSE 的 `data:` 行返回；只支持一种格式会被误判为无结果。

**根因**

将 MCP 当成普通 REST 搜索接口，没有实现协议生命周期和内容类型兼容。

**解决**

- 实现 `initialize -> notifications/initialized -> tools/list -> tools/call` 生命周期。
- 协商并发送 MCP 协议版本。
- 同时解析 JSON 与 SSE 响应，保留会话标识。
- 将工具返回的标题、URL 和摘要规范化为可引用网页来源。
- 对网络错误、无工具、协议错误和无 URL 结果使用明确异常。

**验证与预防**

使用本地伪 MCP 服务覆盖完整生命周期和两种响应格式。替换搜索提供商时先确认工具参数和返回结构，不在业务层硬编码特定厂商字段。

## 20. “发送后像卡住”与真正的端到端流式输出

**现象**

非流式接口要等待课程检索、联网判断和模型生成全部完成后才返回。用户点击发送后长时间看不到变化，会认为页面卡住。仅在前端加转圈可以缓解焦虑，但不能展示模型正在生成的内容。

**根因**

原后端调用 OpenAI-compatible 接口时使用普通 JSON 响应，Python 服务也只在完整答案生成后写回。后来即使增加 NDJSON 和 `flush()`，HTTP/1.0 连接与浏览器缓冲仍可能让小首包延迟交付。

**解决**

- 前端点击发送后立即插入用户消息和空的助手占位消息，显示带脉冲动画的“正在发送…”。
- 后端发送接收、附件、ReAct 思考、课程检索、联网、LLM 重试和回答生成状态，并用 `thought` 事件展示 planner action。
- LLM 请求启用 `stream: true`，逐行解析 OpenAI SSE `data:` delta。
- 浏览器使用 `ReadableStream` 和增量 `TextDecoder` 解析 SSE `data:` 帧。
- SSE 消费回调支持异步背压；即使一个网络分块包含多个 token，也会逐个等待 UI 绘制，避免 Vue 在同一任务中批量合并更新。
- 服务升级为 HTTP/1.1，并使用标准 `Transfer-Encoding: chunked` 对每个事件分块、立即刷新，最后发送零长度结束块。
- 新前端请求 `text/event-stream`；后端按 `Accept` 做内容协商，仍为已经打开的旧页面返回 NDJSON，避免升级瞬间出现 `data:` JSON 解析错误。旧页面刷新后即切换到 SSE 与逐 token 绘制。
- 不支持流式的模型可使用普通生成；瞬时请求错误最多共尝试五次，并通过 `llm_retry` 展示进度。
- 已经输出 token 后不重放请求，避免重复内容；持续失败作为请求错误返回。
- 用户主动停止或流式中断时保留已经显示的文字，不把半段内容静默清空。

**没有采用的方案**

曾尝试给首个状态事件添加 2KB、8KB 空白以越过浏览器缓冲阈值。该方案浪费带宽且依赖浏览器实现，最终改为标准 HTTP 分块传输。

**验证与预防**

测试模型 delta、SSE 跨块解析、异步逐 token 消费、HTTP 分块帧、立即占位、状态变化、附件流式请求、结束事件和失败回退。真实环境还需观察发送后占位是否立即出现、回答是否逐步增长、引用是否在结束后完整显示。

## 21. 图片、PDF 和 DOCX 的解析能力不一致

**现象**

TXT/Markdown 可直接读取，PDF 可能是文本型或扫描型，图片需要视觉模型，DOCX 只能提取基础正文。若把所有格式都当普通文本，会出现空索引或误导性回答。

**根因**

文件格式的内容提取成本不同，且 MinerU、视觉模型均属于可选外部能力。

**解决**

- 文本和 Markdown 使用本地读取。
- PDF 优先调用 MinerU API/命令，失败后用 `pypdf` 处理文本型 PDF。
- 聊天图片转为 OpenAI-compatible `image_url` 内容块，仅在模型支持时进行视觉理解。
- DOCX 使用标准库读取压缩包中的正文 XML；损坏文件返回明确失败占位，复杂版式仍建议转换为 PDF/TXT。
- 可选解析服务失败时给出质量或降级说明，不阻断其他课程文件。

**验证与预防**

测试 MinerU 自动发现、显式命令、API 鉴权、禁用状态、图片上传和安全文件名。增加新格式前先定义本地、外部和失败三条路径。

## 22. 上传文件可能包含路径穿越或重名风险

**现象**

浏览器上传的文件名理论上可以包含目录分隔符；直接拼接到目标目录可能写出课程目录。重名文件也可能覆盖已有资料。

**根因**

客户端文件名不能被当作可信路径。

**解决**

- 服务端统一提取安全文件名，移除路径片段和危险字符。
- 课程上传和聊天附件分别写入受控目录。
- 仅允许支持的扩展名进入课程文件树与解析流程。
- 静态文件服务解析真实路径并检查必须位于构建目录内。

**验证与预防**

覆盖 Windows/Unix 路径片段、非法扩展名、目录穿越和静态资源越界测试。所有文件系统入口都必须在服务端再次校验。

## 23. 本地端口占用、权限和网络失败难以区分

**现象**

启动失败可能来自 8000 端口已被占用、运行环境禁止绑定端口、外部 API 无网络、DNS 失败或代理配置，而用户往往只看到“连接失败”。

**根因**

本地监听、浏览器访问和外部服务访问属于三条不同网络路径，不能用同一个错误结论解释。

**解决**

- 启动日志明确输出本地 URL。
- 分别检查本地首页、`/api/config` 和外部服务状态。
- 本地端口问题先查占用与权限；外部模型/MCP 问题查看处理轨迹中的 `fallback` 或失败状态。
- 外部服务不可用时保留本地检索能力。

**验证与预防**

发布前检查首页和配置接口返回 200，再执行一次真实课程问答。排障时不要因为网页能打开就认定模型网络也正常。

## 24. 测试、运行数据和构建产物容易污染提交

**现象**

真实验收问题会写入课程会话和记忆；前端构建产生 `web/dist`，npm 产生 `node_modules`，Python 产生缓存。若直接全部暂存，会提交用户数据、密钥或庞大依赖目录。

**根因**

本地文件型应用的运行数据与源码位于同一工作区，需要明确版本控制边界。

**解决**

- `.gitignore` 排除真实配置、课程记忆、索引、上传、虚拟环境、Node 依赖和构建产物。
- 浏览器验收使用唯一问题，完成后只删除能够精确匹配的最后一对测试消息与对应记忆。
- 提交时显式列出源码、测试和文档文件，不使用不加检查的全目录暂存。
- 提交前执行 `git diff --check` 和 `git status`。

**验证与预防**

确认暂存区不包含 `data/config.json`、`data/course_memory`、`.venv`、`frontend/node_modules` 或 `web/dist`。任何自动化验收都应包含测试数据清理步骤。

## 25. 当前仍保留的限制与后续方向

以下不是未修复缺陷，而是当前课程项目有意保留的边界：

- DOCX 已支持基础正文抽取，但不保留复杂版式、批注、页眉页脚或图片文字。
- 当前已实现 OpenAI-compatible embedding、持久向量索引、hybrid 融合和可选外部 rerank；仍未使用独立向量数据库或 multi-vector 索引。
- 外部 LLM、embedding、rerank、MinerU 和 Web Search 的质量、额度与可用性不由本项目控制。未配置能力有本地边界；已配置 LLM 的持续请求失败会明确报错，而不是保证无损降级。
- 安装脚本负责项目依赖，不自动安装系统 Python/Node。系统运行时应通过官网、包管理器或学校统一环境安装。
- 文件型存储适合单机课程项目，不适合多用户并发服务；扩展到多用户时应引入数据库、鉴权和数据迁移机制。

## 26. 推荐排障顺序

遇到“不能运行”或“回答异常”时，按以下顺序可以最快缩小范围：

1. 运行 `install-deps.bat` 或 `./install-deps.sh`，确认 pip 与 npm 均成功。
2. 检查 Python、Node 和 npm 版本，不要只看是否安装。
3. 运行前端类型检查、测试和构建。
4. 运行后端测试。
5. 启动服务，检查首页和 `/api/config`。
6. 设置课程目录并重新构建当前课程索引。
7. 用课程中有明确答案的问题测试本地引用。
8. 再测试课程没有覆盖的问题，查看 LLM 状态。
9. 最后测试需要最新信息的问题，查看 Web Search 状态与 `[W]` 引用。
10. 若流式体验异常，分别检查前端立即占位、SSE 事件、异步绘制队列、HTTP 分块和上游模型 SSE。

常用验证命令：

```bash
python3 -m unittest discover -s tests -v
npm test --prefix frontend -- --run
npm run typecheck --prefix frontend
npm run build --prefix frontend
git diff --check
git status --short
```

## 27. 超短问题触发无关联网结果，以及如何停止生成

**现象**

学生只输入 `1` 时，本地检索没有可靠依据，旧策略据此自动联网。搜索引擎把 `1` 理解为数字一，返回数学、字典等大量网页，模型随后围绕这些无关资料扩写，看起来像流式输出产生了幻觉。生成期间也没有停止入口，用户只能等待整轮完成。

**根因**

- “本地证据不足”被直接等价为“应该联网”，没有先判断问题是否完整。
- Web Search 摘要上限偏大，多条结果会把无关正文带入 Prompt。
- 前端流请求没有暴露 `AbortSignal`，后端写流失败时只返回布尔值，无法中止上游生成循环。

**解决**

- 增加不完整问题检测：纯数字、单字符和只有标点的问题直接请求补充上下文。
- 澄清分支不联网、不调用 LLM、不写入课程记忆，也不保留偶然命中的课程引用。
- 网页来源摘要从 800 字限制到 360 字，降低无关上下文占比。
- 生成期间发送按钮切换为停止按钮；`AbortController` 立即终止浏览器读取并保留已显示内容。
- 后端遇到断开的流连接后抛出内部取消信号，退出当前生成循环并关闭上游响应。

**验证与预防**

集成测试断言问题 `1` 不调用搜索和模型；前端测试断言停止后信号为 aborted、无错误提示且部分回答仍保留；后端测试断言断开写流会终止生成。后续新增自动工具调用前，都应先增加输入完整性与调用必要性判断。

## 28. Markdown 预览显示为未排版的源码

**现象**

点击 `.md` 或 `.markdown` 文件后，右侧预览区直接显示 `#`、`-` 和代码围栏等 Markdown 源字符，标题、列表、表格、引用和代码块没有形成可阅读的层次。

**根因**

预览组件把所有非 PDF、非图片文件都放进同一个 `<pre>` 元素，没有根据扩展名选择渲染方式。直接使用 `v-html` 拼接 Markdown 也不可取，因为课程文件可能包含原始 HTML，未经清理会产生 XSS 风险。

**解决**

- 仅对 `.md` 和 `.markdown` 使用 `marked` 解析，TXT 继续按原样显示。
- 使用 `DOMPurify` 清理解析结果后再交给 `v-html`，移除脚本和事件处理属性。
- 为标题、列表、表格、引用、行内代码和代码块提供与工作区配色一致的排版样式。
- 将 Markdown 渲染封装为独立服务，使解析和安全策略不散落在组件中。

**验证与预防**

组件测试同时覆盖结构化渲染、危险 HTML 清理和 TXT 原样预览。新增 Markdown 扩展能力时，应继续通过解析器扩展点实现，不要用正则手写 Markdown，也不要把未经清理的 HTML 注入页面。

## 29. 单课程单会话无法支撑不同学习主题

**现象**

同一门课程中的作业讨论、概念复习和考试准备全部写入一份 `messages.json` 与 `memory.md`。历史越长，追问改写和提示披露越容易带入无关上下文，用户也无法整理主题。

**解决**

- 增加 `conversations.json` 和 `conversations/<conversation_id>/` 目录。
- 新建、切换、重命名、删除和标记已读均通过课程级 conversation API。
- ChatFlow、摘要、练习题和清空操作显式接收 `conversation_id`。
- 第一次读取时把旧 `messages.json`/`memory.md` 复制到 `default` 历史对话，保留旧文件用于回滚。

**验证与预防**

Store、HTTP 和前端测试同时覆盖对话隔离。以后新增任何写入消息或 memory 的入口，都必须先回答它属于课程级还是对话级状态。

## 30. 学习模式不能只在答案前拼接固定文案

**现象**

旧模式只在最终回答上添加前缀，工具选择仍完全相同。启发提示可能第一轮直接给答案，复习模式也不能主动组织回忆任务。

**解决**

- 收敛为 `answer`、`guide`、`review` 三种规范模式，兼容旧 `socratic`/`homework`。
- `StudyModePolicy` 同时提供 planner 和 responder 规则。
- `guide` 根据当前对话中的有效尝试逐级披露，`review` 强调结构、回忆和自测。
- planner 最多三轮，把课程检索、联网和澄清作为显式 action，并把 action 通过 `thought` 事件展示。

**验证与预防**

测试模式归一化、prompt 注入、工具决策、本地 fallback 和前端选项。新模式必须同时定义工具策略和回答策略。

## 31. LLM 重试不能重复已经显示的回答

**现象**

网络瞬时中断会让整个聊天失败；如果在流式已经收到 token 后重放请求，又会在界面生成重复段落。

**解决**

- 在尚未收到 token 时对可重试网络错误、408/409/425/429 和 5xx 最多共尝试五次。
- 用 callback 发送 `llm_retry` 状态，避免用户误以为页面卡住。
- 流式已经产生 token 后不重放；错误由上层显示并保留部分内容。

**验证与预防**

LLM client 测试覆盖成功重试、达到上限、不可重试错误和已有 token 的流式失败。文档必须区分“未配置时的本地 fallback”和“已配置 provider 的持续请求失败”。

## 32. 功能原型、后端能力和当前 UI 容易混写

**现象**

学习计划曾有完整原型，Dashboard/mastery 仍有后端 API 和 Pinia actions，但最新侧栏已经不展示这些区域。旧报告和截图却继续把它们标为当前可操作且验收通过。

**解决**

- 从公开路由和前端移除学习计划，保留内部遗留代码但不作为产品能力。
- 明确 Dashboard/mastery 是后端保留能力，当前 UI 不展示。
- 建立 `docs/index.md` 区分当前文档、交付快照和历史设计；截图统一重拍到 `docs/screenshots/`。

**验证与预防**

前端契约测试断言旧区域和 `/plan` 请求不存在。验收材料必须以当前可见 UI 为准；历史日志可以记录曾实现原型，但要注明最终范围调整。
